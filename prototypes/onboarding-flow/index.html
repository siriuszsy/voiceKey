Onboarding Prototype

先让用户在 2 分钟内成功打出第一句话。

这个原型不讨论更多功能，只收口首次引导。目标是把「百炼 API Key」、 「麦克风授权」、「第一次成功听写」和「辅助功能直写」拆成连续的 5 步， 把边界效应压到最低。

主路径
本次使用

默认先不碰钥匙串，也不逼用户做长期配置。

核心目标
第一次成功输入

不是先学设置页，而是先证明音键能出字。

权限顺序
麦克风 → 辅助功能

先拿录音，再引导直写，不把用户卡在前置条件里。

Design Rules

引导里每一步只解决一个问题。

1. 先跑起来

API Key 主路径默认是“本次使用”。

先让用户完成一次体验，再谈安全存储、环境变量和高级配置。

2. 先出字

第一次成功听写必须发生在引导里。

先让用户看到“音键真的能工作”，再继续讲辅助功能和直写。

3. 少解释

不把完整设置页当 onboarding。

首次引导只保留关键决策，把“知道太多”变成“成功太慢”。

首次引导单页原型

左侧是步骤和场景切换，右侧是当前步骤。这个版本故意不用系统设置窗口风格， 而是做成更轻的 onboarding 面板，让“拿 Key、授权、第一次成功输入”看起来是一条完整路径。

建议先看默认场景，再切“已拒绝麦克风”和“不启用辅助功能”

这版收口的问题

不是扩功能，而是降低第一次上手损耗。

主成功路径

欢迎 → API Key → 权限准备 → 直接写入测试

不要一开始就讲

钥匙串、环境变量、realtime/offline 细节、模型名、完整设置项。

要在引导里显式解释的边界

不开麦克风不能录音；不开辅助功能也能用，但第一次测试就应该明确看到它会回退到剪贴板。

当前最大流失点

用户还没完成第一次成功输入，就被 API Key、权限和设置页复杂度劝退。

建议这套首次引导将来单独作为一个窗口或第一页，而不是把现有设置页硬塞更多说明文字。
